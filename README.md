## 레포트 내용

1. 이더리움 솔리디티에서 외부와 상호작용하는 방식

이더리움 솔리디티에서 외부 컨트랙트와 상호작용하는 방법은 크게 두가지가 있다. 하나는 인터페이스를 사용하는 방식이고 다른 하나는 call을 사용하는 방식이다.

인터페이스 방식은 이 컨트랙트에 어떤 함수들이 있고, 어떻게 호출해야 하는 가를 정의해둔 설계도라고 이해하면 된다.

call 방식은 이 컨트랙트의 주소를 알고 있어야하고, 주소만 알고 있어도 해당 컨트랙트를 호출할 수가 있다. call 호출은 단순히 컨트랙트의 함수를 실행하는것을 넘어 이더리움 송금이나 가변적인 데이터 처리까지 가능하다.

인터페이스 호출 방식과 call 호출 방식의 비교점을 간단하게 정리하면

1. 안정성 측면에서는 인터페이스 방식이 좋다.(인터페이스 방식은 컴파일 타임에 타입 체크가능, call 방식은 런타임에 결정)
2. call 방식은 인터페이스와 달리 가스 제어를 개발자가 직접 할 수 있다.
3. 에러처리 측면에서 인터페이스 호출은 실패시 자동 Revert 되지만 call 방식은 반환되는 성공여부(bool값)로 에러 처리를 해야 한다.
4. 인터페이스는 일반적인 컨트랙트간 통신에 주로 사용되고, call은 이더리움 송금 가변적인 데이터 처리에 사용된다.

이렇게 스마트컨트랙트가 외부와 상호작용하는 방식을 간단하게 먼저 정리해보았다.

2. delegatecall

delegatecall은 솔리디티에서 제공하는 기능중 하나이다. 일반적인 call은 호출한 컨트랙트의 함수를 실행시켜달라는 요청이라면, delegatecall은 호출한 컨트랙트의 코드를 빌려와 나의 영역에서 실행할게 라는 요청이라고 이해를 하면 좋을 것 같다.

delegatecall은 Storage를 공유한다는 특징이 있다. A컨트랙트가 B컨트랙트의 함수를 delegatecall로 호출하면 함수는 B컨트랙트에 정의되어 있지만 상태변수는 B를 호출한 A의 저장소에 기록된다.

또한 msg.sender와 msg.value가 변하지 않는다. B컨트랙트 입장에서는 A컨트랙트를 거쳐왔음에도, 원래 A컨트랙트를 호출했던 최초 사용자를 msg.sender로 인식한다.

이렇게 호출된 B컨트랙트는 그저 로직만 제공할 뿐이다.

Storage를 공유한다는 점을 설명하였는데, 여기서 솔리디티에서 가장 빈번하고 치명적인 사고로 이어질 수 있는 부분이 있다. 저장소 충돌(Storage Collision)이다.

먼저 EVM이 데이터를 어떻게 저장하는지 간단하게 설명하면, EVM의 데이터 저장방식은 슬롯(Slot)이다. EVM은 상태변수를 변수명으로 구분하는 것이 아니다. 대신 0번부터 시작하는 32바이트 크기의 '슬롯'이라는 공간에 순서대로 차곡차곡 쌓아간다.

첫번째 선언된 변수 -> Slot 0
두번째 선언된 변수 -> Slot 1

(이후 순차적으로 할당..)

저장소 충돌에 대한 위험을 설명하면, delegatecall은 피호출자(로직제공자)의 코드를 가져와 호출자(프록시)의 저장소 슬롯을 건드리면서 발생하게 된다. 이때 두 컨트랙트의 변수 구성(layout)이 다르면 엉뚱한 데이터를 덮어 쓸 위험이 있다.

예시상황

1. 프록시 컨트랙트 : Slot 0에 address Myaddrress를 선언하고 주소를 저장한다.
2. 로직 컨트랙트 : Slot 0에 uint256 Mynumber 변수를 선언하고 저장한다.
3. 실행 : 프록시가 로직의 setNumber(10) 함수를 delegatecall을 한다.
4. 결과 : 로직 컨트랙트의 코드는 'Slot 0에 10을 저장하라' 라고 명령하게 되고, 결국 프록시 컨트랙트의 Slot 0에 있던 Myaddress 주소값이 10으로 덮어씌워진다.

이렇게 되면 프록시는 주소값을 잃어버려 컨트랙트가 고장나거나, 다른 방식으로 해커가 자신의 주소를 덮어싀워 권한을 탈취할 수 있다는 위험성이 있다.

이러한 위험성이 있지만, delegatecall은 컨트랙트를 업그레이드 가능하게 만들었다는 강한 이점이 있다.

3. 스마트컨트랙트의 업그레이드

블록체인에 배포된 스마트 컨트랙트는 수정이 불가능하지만, 이 delegatecall을 사용해 로직을 업데이트할 수 있는 구조를 만들 수 있게 되었다.

프록시 컨트랙트 : 모든 데이터(잔액, 권한등)를 가지고 있으며 사용자가 호출하는 주소이다.
실행 컨트랙트 : 실제 실행될 로직(함수)만 들어있다.

프록시는 실행 컨트랙트의 주소를 저장하고 해당 주소를 delegatecall로 호출하여 로직을 사용한다. 만약 로직에 버그가 있거나 추가 수정을 해야 한다면 프록시 컨트랙트가 저장하고 있는 실행 컨트랙트의 주소를 새로운 버전의 실행 컨트랙트 주소로 update를 하기만 하면 된다. 그렇게 되면 사용자는 항상 프록시 컨트랙트와만 상호작용하고 데이터는 유지된 채 로직만 바뀌는(버전 업그레이드) 효과를 얻을 수 있다.

4. 당행의 코어뱅킹 형상관리(업그레이드) 방식과 delegatecall방식의 업그레이드의 비교

- 버전관리 주체

  현재 당행 코어뱅킹은 svn 방식(소스 코드 히스토리 관리 방식)의 소스코드 관리 방식을 채택하고 있다. svn방식은 중앙서버(Repository)가 모든 버전의 코드를 저장해두고 있다. 사용자는 특정 리비전(Revision)을 체크아웃(Chekcout)하여 로컬 환경에서 실행한다.

  delegatecall은 프록시 컨트랙트가 이 중앙서버 역할을 한다. 사용자는 항상 프록시 컨트랙트의 주소로만 접근하며, 프록시가 어떤 실행 컨트랙트를 바라보냐에 따라 실행되는 버전이 결정된다.

- 업데이트 방식

  svn은 클라이언트가 svn 업데이트를 명령한다. 소스코드를 내려 받아 빌드/실행/배포 하면서 소스 버전이 업그레이드 된다.

  delegatecall은 클라이언트가 할 일은 없다. 운영자가 프록시 컨트랙트가 가리키는 실행 컨트랙트의 주소만 업데이트 하면 다음 호출부터 즉시 새 버전의 로직이 적용된다.

- 롤백 메커니즘

  svn은 특정 리비전으로 회귀한 후 코드를 다시 빌드/실행/배포 해야 한다.

  delegatecall은 프록시컨트랙트가 바라보는 주소를 이전 버전의 실행 컨트랙트 주소로 다시 업데이트하기만 하면 즉시 롤백이 가느아하다. 거의 실시간 롤백이 가능한 것이다.

- 데이터 유지

  svn은 코드와 데이터베이스가 분리되어 있어 버전을 업그레이드 하면서 종속된 데이터베이스 스키마변경 작업이나 마이그레이션 작업이 별도로 필요하다.

  delegatecall은 데이터가 컨트랙트에 고정되어 있기에 변수 Slot 구조만 잘 유지한다면 별도의 데이터 작업 없이 새로운 기능을 즉시 배포할 수 있다.

5. 코어뱅킹 관점에서 본 delegatecall의 위험성

- svn방식에서는 잘못된 코드를 배포하여도 로직의 오류로 발생하는 문제 수준에서 코드를 수정하여 빌드/실행/재배포를 하면 되지만, delegatecall 방식의 경우 로직의 오류가 그대로 컨트랙트의 Storage 영역, 즉 데이터 영역에 직접 노출된다는 큰 위험성이 있다. 데이터 슬롯을 잘못 사용하게 될 경우 자산이 없어지거나 사용 불가능한 컨트랙트가 될 가능성이 존재한다.

정리하면, delgatecall을 통한 업그레이드는 svn처럼 코드를 관리하는 방식이라기 보다는 배포되어있는 환경에서 서비스 중단 없이 기능을 교체하는 운영기술 (코어 백엔드 개발자로서는 무중단 배포와 매우 유사하게 느껴진다)에 가깝다. 형상관리 단계에서 완벽하게 검증된 코드만이 프록시 컨트랙트에 연결된 자격을 얻어야 한다.
